The Three CAC Certificates
PIV Authentication Certificate
Used for card-based login (TLS client auth). Key Usage: Digital Signature. The private key never leaves the card's hardware security module.
Email Signing Certificate
Used for S/MIME digital signatures on .mil email. Non-Repudiation key usage. Required for classified email on SIPRNet/NIPRNet.
Key Management Certificate
Used for email encryption (key escrow). Backed up by Key Recovery Authority (KRA) — the only CAC cert with escrowed private key.

How CAC/PKI Authentication Works

The full authentication handshake — from card insertion to session establishment.

1
Card Detection & ATR
PC/SC middleware detects card insertion. The card transmits an Answer-To-Reset (ATR) byte sequence identifying its type (T=0 or T=1 protocol), historical bytes, and capability flags. The host selects the PKCS#15 or PIV applet via an APDU SELECT command.
2
Certificate Enumeration
The host reads the card's X.509 certificates from dedicated EF (Elementary File) containers. For PIV cards, certificates live in specific data objects (9A PIV Auth, 9C Signing, 9D Key Mgmt, 9E Card Auth). The public keys are extracted for the browser's cert picker.
3
PIN Verification
The PIN is sent to the card's secure element as an ISO 7816-4 VERIFY APDU. The card validates it internally — the PIN never leaves the card. If the card's retry counter reaches zero, the VERIFY command returns 0x6983 (Authentication Method Blocked).
4
Challenge-Response & TLS
In mutual TLS (mTLS), the server sends a TLS CertificateRequest. The browser sends a SIGN APDU to the card, which produces an ECDSA signature over the handshake hash. The server verifies the signature using the public key from the card's PIV Auth cert, completing mutual authentication.
Key APDU Commands (ISO 7816-4)
00 A4 04 00 09 A0 00 00 03 08 00 00 10 00
→ SELECT PIV Application (AID)
00 20 00 80 08 [PIN padded to 8]
→ VERIFY PIN (reference data 0x80)
00 87 11 9A [Lc] [challenge hash bytes]
→ GENERAL AUTHENTICATE (key ref 0x9A)

⚠ Simulated vs. Production
This demo generates an ephemeral ECDSA key pair in the browser on each run. In production, the key lives in the CAC's hardware secure element and can never be extracted. The cryptographic operations (signing, verification) are real — the key material is not persistent.
✓ Production Requirements
Real CAC auth requires: server-side mTLS (Nginx/Apache with ssl_verify_client on), OCSP/CRL validation against DoD CA endpoints, LDAP lookup against DEERS, and DoD-issued server certificate. GitHub Pages cannot host this — requires a DoD-approved hosting environment.
