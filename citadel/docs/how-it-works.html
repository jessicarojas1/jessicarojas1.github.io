How CITADEL Works

The end-to-end process, step-by-step usage, how to read the results, and how to run it in CI/CD and the cloud.

1. What CITADEL is

CITADEL (Code Inspection, Threat Analysis & Deployment Evaluation Lab) is an enterprise source-code, executable and script review platform. You give it code (or a binary); it tells you how secure it is, how well it's written, what languages it uses, how it's deployed, and which compliance obligations it touches — in one report, mapped to the exact controls of 20+ frameworks.

The public app runs 100% in your browser — nothing is uploaded. An optional deep-scan backend adds real open-source scanners (Semgrep, Trivy, Grype, Syft, Gitleaks, Bandit, ClamAV) and AI-assisted fixes.

2. The process (what happens to your code)

  Ingest → Classify → Analyze (×6) → Score → Map to controls → Report
   files     language    SAST · secrets · SBOM/CVE · binary · quality · deploy
1
Ingest. Files, folders, ZIP/JAR/APK archives (expanded in-browser) and executables are normalized into a flat list. Build/vendor dirs (node_modules, .git…) are skipped; each file is sniffed as text or binary.
2
Classify. Every file is mapped to one of 117 languages/formats by extension and filename, with a language breakdown.
3
SAST. 100+ heuristic rules (regex, language-aware) flag injection, XSS, broken crypto, deserialization, SSRF, path traversal, XXE, insecure config and IaC misconfig. Deep scan adds Semgrep + Bandit.
4
Secrets. Shannon-entropy + pattern detection surfaces hardcoded keys, tokens, private keys and passwords. Deep scan adds Gitleaks + Trivy.
5
SBOM & CVEs. Dependency manifests (npm, PyPI, Maven, Go, Gem, Composer, Cargo, NuGet) become a CycloneDX SBOM, cross-checked against the OSV.dev advisory database for real CVEs (Syft/Grype/Trivy on the backend).
6
Binary & quality. Executables/bytecode are inspected (format, entropy/packing, strings, capability indicators). Quality computes a maintainability index, comment density, licenses, and deployment/IaC detection.
7
Score. Findings are severity-weighted and volume-normalized into a security score and an A–F grade.
8
Map & report. Every finding is cross-walked to the specific controls it implicates across all frameworks, then rendered into the report.

3. Running a scan (step by step)

  1. Open the analyzer (CITADEL home).
  2. Choose what to analyze:
    • Load demo project — instant synthetic vulnerable app to see the tool in action.
    • Select files or Select folder — pick source files or a whole project.
    • Drag & drop a .zip, folder, source file, or executable onto the dropzone.
    • Scan a public repo by URL (deep-scan backend only) — paste a GitHub/GitLab URL.
  3. Watch the progress bar move through the pipeline stages.
  4. The results section appears with the Report tab open. Quick scans then enrich dependencies with live OSV CVEs and refresh automatically.

Files never leave your browser in the client-side app. In deep-scan mode the upload goes to your own backend, which only reads the files (never executes them) and deletes them after the scan.

5. Reading the report

After a scan the results area shows a scorecard (grade + scores) and tabs:

  • Report — the executive summary: verdict, grade, vulnerabilities & findings, risk hotspots (riskiest files), compliance posture, dependencies/CVEs, licenses, deployment. Download full report gives a standalone HTML file.
  • Overview — charts: severity, language composition, weakness categories.
  • Findings — every finding, filterable by severity. Expand one for location, code snippet, fix, mapped controls, and (backend) "Explain & fix".
  • Compliance — per-framework posture with "controls implicated / total" and a View all N controls expander (implicated controls flagged).
  • SBOM — dependencies, ecosystems, known CVEs, and licenses.
  • Binaries — format, entropy/packing, capability indicators, embedded URLs.
  • Quality — maintainability metrics and interpretation.
  • Deployment — detected Docker/K8s/Terraform/CI signals.
  • AI Fix Prompt — see section 6.
  • History — past scans (local), trend, and run-to-run comparison.
  • Export — see section 9.

6. Fixing the issues with AI

  1. Open the AI Fix Prompt tab.
  2. Click Copy prompt (or Download). It contains exact, structured wording listing every finding — location, flagged code, the required fix, and the compliance controls affected — instructing the assistant to fix them in priority order.
  3. Paste it into Claude (or any coding assistant) together with your code, and apply the returned diffs.
  4. Re-scan to confirm the grade improved and no regressions were introduced.

With the backend configured (an ANTHROPIC_API_KEY), each finding also gets an inline Explain & fix button powered by Claude.

7. Triage workflow

  • Prioritize by severity and the Risk hotspots list — fix critical/high first.
  • Accept risk: on any finding, click "Accept risk" to suppress it (stored locally). Toggle "Show suppressed" to review accepted items.
  • Track progress: every scan is recorded in History; use Compare to see security/findings deltas between two runs (green = improvement).

8. How findings map to controls

Each finding has a category (e.g. injection, broken crypto, secrets). A cross-walk maps every category to the specific controls it implicates in each framework — so one SQL-injection finding simultaneously flags OWASP A03, CWE-89, NIST 800-53 SI-10, NIST 800-171 3.14.1, CMMC SI.L1-3.14.1, ISO 27001 A.8.28, PCI 6.2.4, and so on. The Compliance tab shows the full control catalog per framework with implicated controls highlighted, and the Export tab produces POA&M and SSP appendix documents from this mapping.

9. Exports & CI/CD integration

From the Export tab (or Report): JSON, SARIF 2.1.0 (GitHub Code Scanning), CycloneDX SBOM, POA&M (CSV), SSP control appendix (Markdown), JUnit XML (CI test reports), PR-comment Markdown, summary Markdown, and printable PDF.

Run it in CI

# GitHub Action (uploads SARIF to code scanning, fails build on high+)
- uses: ./citadel            # action.yml
  with: { path: '.', fail-on: 'high' }

# Or the CLI directly (inside the Docker image)
node citadel/server/cli.js ./src --format sarif --fail-on high

See CI.md for the full GitHub Action + CLI reference.

10. Deploying CITADEL

Static (quick scan only)

The browser app is already live on GitHub Pages. Any static host works.

Deep-scan backend (Docker)

# build from the repo root (bundles the SPA + all scanners)
docker build -f citadel/server/Dockerfile -t citadel-server .
docker run -p 8080:8080 citadel-server      # http://localhost:8080/

Render

New → Web Service → Docker → Dockerfile ./citadel/server/Dockerfile, build context ., port 8080, Standard (2 GB) instance, health check /api/health. Optional ANTHROPIC_API_KEY for AI fixes. (A one-click Blueprint exists in render.yaml.)

Government cloud

Hardened IaC ships under citadel/deploy/azure-gov (Bicep) and citadel/deploy/aws-gov (Terraform). Use scripts/deploy.sh aws|azure to orchestrate, then run scripts/init.sh inside the container to hydrate the Trivy/Grype/ClamAV databases.

11. Roles & recommended workflow

Developer

Scan your branch before a PR → fix critical/high from the Report & AI Fix Prompt → re-scan to confirm grade.

Security reviewer

Use deep scan for full CVE/secret coverage; triage with suppressions; export SARIF to code scanning.

Compliance / GRC

Use the Compliance tab and export POA&M + SSP appendix to seed an ATO package against your framework.

CI/CD

Run the GitHub Action / CLI on every PR; gate merges with --fail-on; post the PR-comment Markdown.

12. Scope & limitations

The browser engine is heuristic (pattern-based) for fast, transparent triage across all 117 languages. The deep-scan backend adds data-flow-grade scanners for the languages those tools support. Neither replaces a credentialed penetration test or a qualified assessor's review. For an Authorization to Operate (ATO), pair CITADEL's output with the backend scanners and human review. Treat all findings as candidates to verify, not verdicts.
