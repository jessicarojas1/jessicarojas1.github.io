1. Purpose

This Cryptography Policy establishes the requirements for the selection, implementation, management, and use of cryptographic controls within the organization. Cryptography is a critical tool for protecting the confidentiality, integrity, and authenticity of organizational information, and for supporting secure communications and digital identity assurance.

The purpose of this policy is to ensure that cryptographic measures are applied consistently and effectively, using approved and current algorithms and key lengths, and that cryptographic keys and certificates are managed in a secure and auditable manner. Poor cryptographic practices — including use of weak algorithms, improper key management, or failure to apply encryption where required — can render security controls ineffective and expose the organization to significant risk.

This policy applies in support of ISO/IEC 27001:2022 Control A.8.24 (Use of cryptography) and complements the requirements of the Data Classification Policy (POL-005) and Access Control Policy (POL-003).

2. Scope

This policy applies to all cryptographic controls used to protect organizational information, including but not limited to:

  • Encryption of data at rest on servers, workstations, mobile devices, databases, and removable media
  • Encryption of data in transit across internal and external networks
  • Digital signatures and certificate-based authentication
  • Cryptographic hashing for data integrity verification and password storage
  • PKI (Public Key Infrastructure) and certificate management
  • Key management systems, Hardware Security Modules (HSMs), and secrets management solutions
  • VPN, SSH, and TLS implementations
  • Encryption used within internally developed or procured applications

This policy applies to all personnel responsible for the design, implementation, operation, or management of systems using cryptographic controls, including IT staff, developers, system administrators, and vendors implementing cryptographic solutions on behalf of the organization.

3. Cryptographic Standards

Only approved cryptographic algorithms and key lengths as defined in this section may be used in organizational systems. Use of algorithms or configurations not listed in this section requires prior written approval from the Information Security Manager. The organization will review and update approved algorithms as the cryptographic landscape evolves.

Password hashing: Passwords must never be stored using general-purpose hash functions (SHA-256, SHA-512, etc.). Password hashing must use a purpose-built, slow key-derivation function: Argon2id (preferred), bcrypt (work factor 12 or higher), or PBKDF2 (600,000+ iterations with HMAC-SHA-256). Plain text storage or reversible encryption of passwords is strictly prohibited.

3.4 Transport Layer Security (TLS)

For TLS 1.2, only strong cipher suites may be used. Cipher suites using NULL encryption, anonymous authentication, export-grade encryption, RC4, or CBC-based cipher suites vulnerable to BEAST/POODLE attacks must be disabled. Approved TLS 1.2 cipher suites include those in the ECDHE family with AES-GCM. The IT department must maintain and enforce a current list of approved cipher suites.

4. Encryption Requirements

The following encryption requirements apply based on data classification and use case:

4.1 Data at Rest

  • All data classified as Confidential or Restricted must be encrypted at rest using AES-256 or AES-128 (minimum)
  • Full-disk encryption must be enabled on all laptops, workstations, and mobile devices using BitLocker (Windows), FileVault (macOS), or equivalent platform-native solution
  • Database encryption (Transparent Data Encryption or column-level encryption) must be applied to databases containing Confidential or Restricted data
  • Backup data containing Confidential or Restricted information must be encrypted prior to storage or transmission
  • Removable media (USB drives, portable hard drives) used to store organizational data must be encrypted

4.2 Data in Transit

  • All data transmitted across untrusted networks (including the internet, public Wi-Fi, and third-party networks) must be encrypted using TLS 1.2 or higher
  • Unencrypted protocols (HTTP, FTP, Telnet, SMTP without STARTTLS) must not be used for transmitting organizational data outside of isolated lab environments
  • All API communications must use HTTPS (TLS 1.2+); HTTP-only APIs are prohibited in production environments
  • All remote administrative access must use encrypted protocols (SSH, RDP with NLA and TLS, HTTPS management interfaces); Telnet and unencrypted HTTP management interfaces are prohibited
  • Email containing Confidential or Restricted content must be protected using S/MIME or PGP encryption, or transmitted via an encrypted secure file transfer mechanism

4.3 Removable Media

Removable media (USB storage, external hard drives, optical media) used to store or transport organizational data must be encrypted using AES-256. Unencrypted removable media containing Confidential or Restricted data is prohibited. The use of removable media for Restricted data must be individually authorized by the data owner and Information Security Manager, and must be tracked and logged.

5. Key Management

Cryptographic keys are as sensitive as the data they protect. Compromise of a key can render encryption ineffective. Key management must be treated as a critical security function.

5.1 Key Generation

  • All cryptographic keys must be generated using a Cryptographically Secure Pseudo-Random Number Generator (CSPRNG). Deterministic or predictable key generation is prohibited.
  • Key generation must occur within a trusted and controlled environment. Where possible, key generation must be performed within an HSM or approved Key Management Service (KMS).
  • Key lengths must comply with the approved algorithm standards in Section 3.

5.2 Key Storage and Protection

  • Cryptographic keys must be stored securely and separately from the data they protect. Keys must not be stored in the same location or system as the encrypted data.
  • Production encryption keys must be stored in an approved Hardware Security Module (HSM) or an approved cloud-native Key Management Service (KMS) such as AWS KMS, Azure Key Vault, or Google Cloud KMS.
  • Keys must never be stored in source code, configuration files, scripts, or environment variables without the use of a secrets management system (e.g., HashiCorp Vault, AWS Secrets Manager).
  • Access to key management systems must be restricted to authorized personnel only and subject to the access controls defined in the Access Control Policy (POL-003), including MFA.
  • All key management operations (creation, rotation, access, deletion) must be logged in an immutable audit trail.

5.3 Key Rotation

Key TypeMaximum Rotation PeriodNotes
Symmetric encryption keys (data at rest)1 yearRotate immediately upon suspected compromise
Asymmetric keys (RSA, ECDSA)2 yearsAligned with certificate lifecycle where applicable

Keys must be rotated immediately, regardless of scheduled rotation timelines, when: a key is known or suspected to have been compromised; personnel with access to the key leave the organization; the algorithm or key length is found to be inadequate; or a security incident involving the system protected by the key occurs.

5.4 Key Escrow and Recovery

Where required for business continuity purposes, cryptographic keys may be escrowed (a copy held securely by a designated custodian) to enable data recovery in the event of key loss. Key escrow must be:

  • Formally authorized and documented, with the escrow arrangement recorded in the key management inventory
  • Protected with at least the same level of security as the primary key
  • Subject to split custody where technically feasible (requiring two authorized parties to reconstruct the key)
  • Tested for recoverability at least annually

Key escrow for keys protecting Restricted data requires approval from the Information Security Manager and must involve at minimum two designated custodians.

6. Certificate Management

Digital certificates are used to bind cryptographic keys to identities and to establish trust in TLS communications, code signing, email security, and authentication. Certificate management must be rigorous to prevent expired, mis-issued, or compromised certificates from undermining security.

  • Only certificates issued by approved, trusted Certificate Authorities (CAs) may be used in production systems. A list of approved CAs must be maintained by the IT department. Unauthorized or unapproved CAs must not be added to system trust stores.
  • Self-signed certificates must not be used in production environments for systems accessible by end users or external parties. Self-signed certificates may only be used in isolated development or test environments, and must never be trusted in production trust stores.
  • All TLS certificates used in production must have a maximum validity period of 1 year (or less, consistent with CA/Browser Forum requirements). Certificate validity periods must not exceed the maximum permitted by the issuing CA's policies.
  • A complete inventory of all organizational certificates must be maintained, including: common name, issuing CA, validity period, responsible owner, and the system(s) on which the certificate is deployed. This inventory must be reviewed at least quarterly.
  • Certificate expiry monitoring must be implemented to provide alerts at minimum 60 days, 30 days, and 7 days before expiration. Automated certificate renewal (e.g., via ACME/Let's Encrypt or enterprise CA automation) must be used where technically feasible.
  • Expired certificates must not remain deployed on production systems. Deployment of an expired certificate is a security control failure that must be treated as an incident.
  • Certificate revocation must be performed promptly when a certificate is compromised, when the associated private key is suspected of exposure, or when the certificate is no longer required. CRL (Certificate Revocation List) or OCSP (Online Certificate Status Protocol) must be verified by systems relying on certificates.

7. Prohibited Practices

The following cryptographic practices are strictly prohibited within the organization:

  • Use of weak or broken algorithms: MD5, SHA-1 (for security purposes), DES, 3DES, RC4, SSL 2.0, SSL 3.0, TLS 1.0, and TLS 1.1 must not be used in any security-sensitive context. Legacy systems still using these algorithms must be prioritized for remediation.
  • Hardcoded cryptographic keys: Embedding cryptographic keys, passwords, API secrets, or tokens directly in source code, configuration files, or compiled binaries is strictly prohibited. Secrets must be stored and retrieved from an approved secrets management solution.
  • Self-signed certificates in production: Self-signed certificates must not be deployed on production systems accessible to users or external systems. Their use undermines certificate-based trust and cannot be effectively revoked.
  • Key sharing: Private keys must not be shared between systems, individuals, or organizations except through formally approved and documented key escrow arrangements. A private key compromise that requires notification or remediation must be treated as a security incident.
  • Homegrown cryptography: Implementing custom or proprietary cryptographic algorithms or protocols is prohibited. Only well-vetted, publicly reviewed, and standards-based algorithms may be used. "Security through obscurity" is not an acceptable substitute for proven cryptographic strength.
  • ECB mode encryption: AES or any block cipher operated in Electronic Codebook (ECB) mode must not be used for data encryption, as ECB mode reveals patterns in plaintext.
  • Key storage in plaintext: Cryptographic keys must never be stored in plaintext form in files, databases, email, or any system that does not provide appropriate access control and protection.

8. Cryptographic Module Standards

The organization prefers the use of cryptographic modules that have been validated under FIPS 140-2 or FIPS 140-3 (Federal Information Processing Standard), as this validation provides assurance that the module correctly implements approved algorithms and key management practices.

FIPS 140-2/140-3 validated modules at Security Level 2 or higher are required for:

  • HSMs used for key generation, storage, and management
  • Systems processing Restricted data where validated modules are technically feasible
  • Any system subject to regulatory requirements mandating FIPS-validated cryptography (e.g., FedRAMP, certain financial or healthcare regulations)

Where FIPS-validated modules are not available or technically feasible, the Information Security Manager must document the deviation and confirm that the cryptographic implementation otherwise complies with the algorithm and key length requirements of this policy. The use of platform-native cryptographic libraries (OpenSSL in FIPS mode, Windows CNG, Apple Security Framework) is generally acceptable as these are regularly updated and reviewed.

Vendors supplying cryptographic solutions must be able to demonstrate the validation status of their cryptographic modules upon request. The validation status of modules in use must be periodically reviewed to ensure they remain current and have not been revoked.
