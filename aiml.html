AI vs Machine Learning — What's the Difference?

These terms are often used interchangeably but they are not the same thing. Understanding the distinction prevents confusion when choosing tools or reading research.

Artificial Intelligence (AI)

The broadest term. Any technique that enables machines to simulate human-like intelligence — reasoning, problem-solving, planning, language understanding, perception.

Includes:

  • Rule-based expert systems (no learning)
  • Search algorithms (A*, minimax)
  • Machine Learning (learns from data)
  • Robotics and planning
  • Natural language processing
AI does not require learning from data. A chess engine with hard-coded rules is AI but not ML.
Machine Learning (ML)

A subset of AI where systems learn patterns from data without being explicitly programmed for every case. The model improves as it sees more examples.

Core idea:

Given input X and output Y, find a function f such that f(X) ≈ Y — and generalise to unseen X.

Three families:

  • Regression — predict a continuous value
  • Classification — predict a category
  • Clustering / Generation — discover structure
Deep Learning (DL)

A subset of ML using multi-layer neural networks with many parameters (millions to billions). Excels on unstructured data: images, audio, text, video.

Why "deep"?

Multiple stacked layers learn increasingly abstract representations — edges → shapes → objects → concepts.

When to use DL over classical ML:

  • Input is images, audio, or raw text
  • You have >100k training examples
  • Feature engineering would be impractical
  • State-of-the-art accuracy required

Types of Machine Learning

Categorised by how the model learns from data.

Supervised Labelled data

Every training example has an input X and a known correct output y. The model learns to map X → y and is evaluated on how well it predicts y on unseen data.

Use when: You have labelled historical data and want to predict a target.

ClassificationRegressionObject detectionTranslation
Unsupervised No labels

Only inputs X, no labels. The model discovers hidden structure — clusters, patterns, compressed representations — purely from the distribution of the data.

Use when: You have lots of data but no labels, or want to explore structure.

Reinforcement Reward signal

An agent takes actions in an environment, receives a reward, and learns a policy that maximises cumulative reward over time. No labelled data — the signal comes from outcomes.

Use when: Sequential decision-making with a clear reward structure.

GamesRoboticsTradingAd bidding
Self-Supervised Labels from data itself

A special form of supervised learning where labels are generated from the input data itself. The model learns by predicting masked/hidden parts of inputs. Powers modern LLMs.

Use when: You have massive unlabelled data and want rich representations.

A small labelled set + a large unlabelled set. The model uses the labelled data for direct supervision and the unlabelled data to learn better representations. Practical when labelling is expensive.

Take a model pretrained on a large general dataset (ImageNet, Common Crawl) and fine-tune it on your smaller task-specific dataset. Dramatically reduces data and compute requirements.

Random Forest Intermediate

An ensemble of decision trees, each trained on a random subset of data and features. Final prediction = majority vote (classification) or mean (regression). Resistant to overfitting; handles missing values.

Key hyperparameters:

  • n_estimators — number of trees (more = better, diminishing returns)
  • max_depth — controls individual tree complexity

When to use: Default choice for tabular data before trying anything more complex.

Gradient boosting builds trees sequentially — each tree corrects the errors of the previous one. Dominates Kaggle competitions on structured data. Faster and more accurate than Random Forest on most tabular problems.

  • n_estimators + early_stopping_rounds
  • max_depth / num_leaves — tree complexity
  • subsample, colsample_bytree — regularization via sampling
Rule: Low learning rate (0.01–0.05) + many trees + early stopping = gold standard.

Regression — From Linear to Neural

Regression predicts a continuous numeric output (price, temperature, score). Understanding the regression family in depth is the foundation of almost all supervised learning, including neural networks.

Linear RegressionBeginner

Fits a straight line (or hyperplane) through the data: ŷ = w₀ + w₁x₁ + w₂x₂ + … + wₙxₙ. Weights are found by minimising the Mean Squared Error (MSE) between predictions and true values.

from sklearn.linear_model import LinearRegression model = LinearRegression() model.fit(X_train, y_train) print(model.coef_) # weight per feature print(model.intercept_) # bias term w₀

Assumptions (when they break, the model fails):

  • Linear relationship between features and target
  • Errors are normally distributed and homoscedastic
  • Features are not perfectly collinear
Ridge & Lasso (Regularised Regression)Beginner

Plain linear regression overfits with many features. Regularisation adds a penalty to large weights, forcing the model to stay simpler.

MethodPenaltyEffect
Ridge (L2)λ · Σwᵢ²Shrinks all weights toward zero; keeps all features
Lasso (L1)λ · Σ|wᵢ|Drives some weights to exactly zero — automatic feature selection
ElasticNetmix of L1+L2Best of both; use when features are correlated
from sklearn.linear_model import Ridge, Lasso, ElasticNet ridge = Ridge(alpha=1.0) # alpha = λ lasso = Lasso(alpha=0.1) enet = ElasticNet(alpha=0.1, l1_ratio=0.5)
Polynomial RegressionBeginner

Fits curves instead of lines by adding polynomial features: x², x³, x·z, etc. Still linear in the weights — just linear regression on transformed features.

from sklearn.preprocessing import PolynomialFeatures from sklearn.pipeline import make_pipeline model = make_pipeline( PolynomialFeatures(degree=3), Ridge(alpha=1.0) # always regularise poly ) model.fit(X_train, y_train)
Degree > 3 almost always overfits badly. Prefer tree-based models or neural networks for complex non-linear relationships.
Logistic RegressionBeginner

Despite the name, logistic regression is a classification model. It applies the sigmoid function to a linear combination of features to output a probability between 0 and 1.

ŷ = σ(w·x + b) where σ(z) = 1/(1 + e⁻ᶻ)

Threshold at 0.5 for binary classification. For multi-class, use softmax (multinomial logistic regression).

  • Fast to train, highly interpretable (odds ratios)
  • Great baseline for any classification problem
  • Fails on non-linear decision boundaries without feature engineering
Regression with Neural NetworksIntermediate

For complex non-linear regression, an MLP replaces the linear model: multiple hidden layers with non-linear activations, then a single linear output neuron (no activation on the last layer for regression).

class MLPRegressor(nn.Module): def __init__(self): super().__init__() self.net = nn.Sequential( nn.Linear(n_features, 128), nn.ReLU(), nn.Linear(128, 64), nn.ReLU(), nn.Linear(64, 1) # linear output ) def forward(self, x): return self.net(x)

Loss: nn.MSELoss() for regression. Use nn.HuberLoss() for robustness to outliers.

K-Means Clustering
Assigns each point to the nearest of K centroids, then recalculates centroids iteratively. Fast, simple, scales well. Assumes spherical clusters of similar size.

Choose K: Elbow method (plot inertia vs K) or Silhouette score.

Density-based clustering. Groups points that are closely packed; marks low-density points as noise/outliers. Doesn't require specifying K; discovers arbitrary-shaped clusters.

Linear dimensionality reduction. Projects data onto orthogonal axes of maximum variance. Use to reduce features, visualise high-dim data, or remove noise before another model.

Neural network families — when to use which.

Neural Network Fundamentals — The MLP Foundation

A Multi-Layer Perceptron (MLP) — also called a fully-connected or feed-forward network — is the simplest neural network. Every advanced architecture (CNN, RNN, Transformer) is built on the same core idea.

Anatomy of a neuron:

output = activation(Σ(weightᵢ × inputᵢ) + bias)

Anatomy of a network:

  • Input layer — one node per feature
  • Hidden layers — learn non-linear representations
  • Output layer — one node per class (classification) or 1 node (regression)

How it learns:

  1. Forward pass: input flows through layers, produces prediction
  2. Loss: compare prediction to truth (MSE, cross-entropy)
  3. Backward pass: gradients flow back via chain rule (backprop)
  4. Update: optimizer adjusts weights to reduce loss
  5. Repeat for every batch until loss converges
import torch.nn as nn # 3-layer MLP: 784 → 256 → 128 → 10 # (e.g., MNIST: 28×28 pixels → 10 digit classes) model = nn.Sequential( nn.Linear(784, 256), # hidden layer 1 nn.ReLU(), nn.Dropout(0.3), nn.Linear(256, 128), # hidden layer 2 nn.ReLU(), nn.Dropout(0.3), nn.Linear(128, 10), # output (logits) ) # Training step logits = model(x_batch) # forward loss = nn.CrossEntropyLoss()(logits, y_batch) loss.backward() # backward optimizer.step() # update

Key design choices:

Activation (hidden)ReLU (or GELU for transformers)
Activation (output)None (regression), Softmax (classification)

CNN — Convolutional Neural Network Intermediate

Applies learned filters (kernels) across input to detect spatial patterns — edges, textures, shapes. Translation-invariant. Standard architecture: Conv layers → ReLU → Pooling → Fully connected.

Key layers: Conv2D, MaxPool, BatchNorm, GlobalAvgPool, Dense

Use for: Images, video frames, audio spectrograms, spatial data

Notable models: ResNet, EfficientNet, VGG, MobileNet, YOLO (detection)

RNN / LSTM / GRU Intermediate

Recurrent networks process sequences by maintaining a hidden state across timesteps. LSTM adds gates (forget/input/output) to control long-range memory. GRU is a lighter variant.

Problem: Vanishing gradients over long sequences — largely replaced by Transformers.

Use for: Time series, legacy NLP, short sequences

Still useful when: Low compute, real-time streaming, short context

Transformer Advanced

Built on self-attention — every token attends to every other token in parallel. No recurrence means it scales with compute. The dominant architecture for language, code, images (ViT), and multimodal models.

Core concepts: Query/Key/Value attention, multi-head attention, positional encoding, feed-forward sublayers, layer norm

Notable models: GPT, BERT, T5, LLaMA, Mistral, Vision Transformer (ViT)

GAN — Generative Adversarial Network Advanced

Two networks in competition: a Generator that creates fake samples and a Discriminator that distinguishes real from fake. Generator improves until the discriminator can't tell the difference.

Training challenge: Mode collapse, training instability, GAN game doesn't always converge

Notable: StyleGAN (faces), CycleGAN (image translation), Pix2Pix

Diffusion Model Advanced

Learns to reverse a noise-adding process. Training: gradually add Gaussian noise to data. Inference: start from pure noise and denoise iteratively. Produces higher quality and more stable output than GANs.

Key concept: DDPM (Denoising Diffusion Probabilistic Model), U-Net backbone, classifier-free guidance

Notable: Stable Diffusion, DALL-E 3, Sora (video), AudioLDM

Graph Neural Network (GNN) Advanced

Operates directly on graph-structured data. Each node aggregates messages from its neighbours, updating its representation. Works on irregular, relational data that CNNs can't handle.

Variants: GCN, GraphSAGE, GAT (attention-based), MPNN

Use for: Social networks, molecules, knowledge graphs, recommendation systems

Reinforcement Learning

Training agents through trial and error in an environment.

Core components: Agent (learner) · Environment (world) · State (current situation) · Action (agent's choice) · Reward (signal) · Policy (action-selection strategy) · Value function (expected future reward)
Q-Learning / DQN Intermediate
Learns a Q-value (expected reward) for each state-action pair. DQN uses a neural network as the Q-function approximator, enabling it to handle high-dimensional state spaces (pixels).
Discrete actionsAtari games
Policy Gradient (REINFORCE) Advanced
Directly optimises the policy (probability distribution over actions) by gradient ascent on expected reward. High variance — needs many samples. Foundation of PPO and modern RLHF.
Continuous actionsLLM fine-tuning (RLHF)
PPO / Actor-Critic Advanced
Proximal Policy Optimization — constrains policy updates to a trust region for stability. Actor-Critic combines value estimation with policy learning. Current standard for RL in practice and RLHF.

Math & Statistics You Actually Need

You don't need a PhD — but these fundamentals are unavoidable.

📐 Linear Algebra

Data is tensors (scalars, vectors, matrices, higher-dim). All neural network operations are matrix multiplications.

  • Vectors: Feature representations, embeddings
  • Matrix multiply: Core of every dense layer (y = Wx + b)
  • Dot product: Similarity between embeddings (cosine similarity)
  • Eigenvalues/vectors: PCA, understanding variance
  • Transpose, inverse: Used throughout gradient derivations
Priority: Matrix multiply > dot product > eigenvalues. You can go far with just the first two.
📉 Calculus & Gradients

Backpropagation is just the chain rule applied repeatedly through a computation graph.

  • Derivative: Rate of change — how much loss changes with each weight
  • Partial derivative: Derivative w.r.t. one variable, holding others fixed
  • Gradient: Vector of all partial derivatives → direction of steepest ascent
  • Chain rule: ∂L/∂w = (∂L/∂a)(∂a/∂w) — propagates error backward
  • Jacobian/Hessian: Higher-order, used in advanced optimizers
Key insight: You don't derive gradients manually — frameworks (PyTorch autograd) do it. But understanding what's happening lets you debug and design better.
🎲 Probability & Statistics
  • Mean, variance, std dev: Describe and normalize your data
  • Distributions: Normal (Gaussian), Bernoulli, Categorical, Beta — model assumptions
  • Bayes' theorem: P(A|B) = P(B|A)P(A)/P(B) — foundation of probabilistic ML
  • MLE vs MAP: Maximum Likelihood Estimation vs Maximum A Posteriori — how models are trained
  • Entropy / KL divergence: Information content, used in cross-entropy loss and VAEs
  • Central Limit Theorem: Why sample means are normally distributed at scale

Training Pipeline

From raw data to a trained model — every step matters.

1 — Data Preparation

Split your data first — before any preprocessing:

  • Train (70–80%): Model learns from this
  • Validation (10–15%): Tune hyperparameters on this
  • Test (10–15%): Final honest evaluation — touch once
Never fit scalers/encoders on validation or test data — only fit on training, then transform all sets. Fitting on all data = data leakage.

Preprocessing steps:

  • Handle missing values (impute mean/median/mode or model-based)
  • Encode categoricals (one-hot, ordinal, target encoding)
  • Scale numerics (StandardScaler for NNs, MinMax optional)
  • Feature engineering (interactions, log transforms, rolling stats)
2 — Forward & Backward Pass

One training step:

  1. Forward pass: Input flows through the network → produces prediction ŷ
  2. Compute loss: Loss(ŷ, y) — how wrong was the prediction
  3. Backward pass: Chain rule propagates gradients from loss back through every layer
  4. Update weights: w ← w − lr × ∂L/∂w
  5. Zero gradients: Reset before next batch (PyTorch doesn't auto-zero)
for epoch in range(num_epochs): for X, y in train_loader: optimizer.zero_grad() # ① reset pred = model(X) # ② forward loss = criterion(pred, y) # ③ loss loss.backward() # ④ backward optimizer.step() # ⑤ update
3 — Batch Size, Epochs & Iterations
EpochOne full pass through the entire training set
Batch sizeSamples per gradient update. Smaller = noisier gradients, better generalisation. Larger = stable, needs more memory.
IterationOne gradient update. Iterations/epoch = dataset_size / batch_size
Mini-batch SGDBatch size 32–512 — standard. Combines efficiency of batch GD with noise benefits of SGD.
Start with: batch_size=32, train until validation loss stops improving (early stopping with patience=10).
4 — Regularization Techniques

Prevent overfitting (model memorises training data, fails on new data):

  • L2 (Weight Decay): Penalises large weights → optimizer = Adam(lr, weight_decay=1e-4)
  • L1: Encourages sparse weights (some exactly zero)
  • Dropout: Randomly zero out neurons during training. Prevents co-adaptation. Rate 0.2–0.5.
  • Batch Normalisation: Normalises activations per batch. Stabilises training, reduces sensitivity to learning rate. Use after linear layer, before activation.
  • Early Stopping: Stop when validation loss stops improving. Most effective regulariser.
  • Data Augmentation: Artificially expand training set (flip, crop, noise).

Optimizers & The Learning Objective

Before choosing an optimizer, understand what it's actually doing — and why that problem is harder than it looks.

The Core Optimization Objective

Every machine learning model — whether linear regression, a random forest, or a billion-parameter transformer — is ultimately solving the same mathematical problem:

minθ  L(θ)
SymbolMeaning
θ (theta)All trainable parameters — every weight and bias in the model
L(θ)Loss function — a scalar measuring how wrong the model's predictions are given the current θ
minFind the θ that makes L as small as possible

For a neural network with millions of parameters, θ is a vector in a millions-dimensional space. The loss surface is a landscape over that space — training is the process of navigating downhill to the lowest point you can find.

Common loss functions (what L(θ) actually is):

TaskLossFormula
RegressionMSE½ · mean((y − ŷ)²)
Regression (robust)HuberMSE for small errors, MAE for large
Binary classificationBinary Cross-Entropy−[y·log(ŷ) + (1−y)·log(1−ŷ)]
Multi-classCross-Entropy−Σ yᵢ · log(ŷᵢ)
RankingHinge Lossmax(0, margin − score)
Generative (VAE)ELBOReconstruction − KL divergence
The loss is not the metric. You minimise the loss (a smooth, differentiable proxy) and evaluate the metric (accuracy, F1, BLEU). Cross-entropy loss minimised → accuracy maximised, but they are not the same number. Always report the metric to stakeholders, never the raw loss.
Gradient Descent — How We Minimise L(θ)

We can't search every possible θ directly — the space is astronomically large. Instead we use the gradient: the direction in θ-space that increases L the most. Move opposite to that direction (downhill) by a small step.

θ ← θ − η · ∇θ L(θ)
SymbolMeaning
η (eta)Learning rate — step size. Too big: overshoots. Too small: slow.
∇θ LGradient of loss w.r.t. parameters — computed by backpropagation
← (update)Applied once per batch (not once per epoch)

This single update rule, repeated thousands to millions of times over mini-batches of data, is how every neural network learns.

Three flavours of gradient descent:

VariantBatch SizeGradient QualitySpeed
Batch GDFull datasetExact — no noiseVery slow; impractical at scale
Stochastic GD (SGD)1 sampleVery noisyFast per step; noisy updates can escape local minima

# PyTorch gradient descent — one update step optimizer.zero_grad() # clear old gradients loss = criterion(model(xb), yb) # forward: compute L(θ) loss.backward() # backward: compute ∇θ L optimizer.step() # update: θ ← θ − η∇θL
zero_grad() is critical. PyTorch accumulates gradients by default. Forgetting to call zero_grad() is one of the most common training bugs — gradients from previous batches pile up and the update is wrong.
Local Minima, Global Minima & Saddle Points

The loss surface of a neural network is not a bowl (convex) — it is a highly complex landscape with many valleys, ridges, and flat regions. Gradient descent can get stuck.

FeatureWhat It IsProblem?
Global minimumLowest possible L(θ) — best solutionRarely found exactly in practice
Local minimumValley — lower than all neighbours, but not globally lowestCan trap gradient descent
Saddle pointFlat region — gradient ≈ 0 but not a minimumMore common than local minima in high dimensions; slows training
PlateauBroad flat area of similar lossLearning appears to stop; often needs LR warmup or schedule change

The surprising truth about local minima in deep learning:

In very high-dimensional parameter spaces (millions of θ), true local minima are extremely rare. A point that is a minimum in every direction simultaneously is vanishingly unlikely. Most apparent local minima are actually saddle points. Research (Goodfellow et al., 2015) showed that local minima in deep nets tend to have similar loss values to the global minimum — so escaping them is less critical than it first appears.

Practical techniques to escape / avoid bad minima:

  • Momentum — accumulates velocity across steps, rolls through small valleys
  • Stochastic noise — mini-batch gradient noise acts as natural perturbation
  • Learning rate warmup — start slow, then accelerate past initial bad regions
  • Cyclical / cosine LR schedules — periodic LR increases help escape plateaus
  • Random restarts — train from multiple random initialisations, take best
  • Good initialisation — Kaiming/He init places starting θ in a better basin
Flat minima generalise better. Sharp minima (narrow valleys) tend to overfit — a tiny shift in test data shoots loss upward. Flat minima are robust to small perturbations. SGD with momentum tends to find flatter minima than Adam, which may explain why SGD sometimes generalises better despite converging slower.

Learning Rate Schedulers

Adjust learning rate during training for faster convergence and better final accuracy.

  • Step Decay: Multiply LR by γ every N epochs
  • Cosine Annealing: Smoothly decays LR following cosine curve — standard for transformers
  • Warmup: Ramp LR from 0 to target over first N steps — critical for large models
  • ReduceLROnPlateau: Reduce when val loss stops improving
  • One-Cycle Policy: LR rises then falls — fast training, often best result

Hyperparameter Tuning

Finding the best configuration for your model — learning rate, batch size, depth, regularization — without manually guessing.

Grid Search Beginner

Exhaustively evaluates every combination of the hyperparameter grid. Simple and reproducible, but cost grows exponentially with the number of parameters.

from sklearn.model_selection import GridSearchCV from sklearn.ensemble import RandomForestClassifier param_grid = { 'n_estimators': [100, 300], 'max_depth': [None, 10, 20], 'min_samples_split': [2, 5] } gs = GridSearchCV(RandomForestClassifier(), param_grid, cv=5, scoring='f1_macro', n_jobs=-1) gs.fit(X_train, y_train) print(gs.best_params_)

When to use: Small grids (<50 combos), final polish after narrowing range.

Random Search Beginner

Samples random combinations from continuous distributions. Finds good solutions 2–10× faster than grid search because most hyperparameters are not equally important.

from sklearn.model_selection import RandomizedSearchCV from scipy.stats import loguniform, randint param_dist = { 'learning_rate': loguniform(1e-4, 1e-1), 'n_estimators': randint(50, 500), 'max_depth': randint(3, 15) } rs = RandomizedSearchCV(model, param_dist, n_iter=60, cv=5, n_jobs=-1) rs.fit(X_train, y_train)

Rule of thumb: 60 random trials achieves ~95% coverage of a well-sampled grid.

Bayesian Optimization Intermediate

Builds a probabilistic surrogate model (Gaussian Process or Tree Parzen Estimator) of the objective surface, then samples where improvement is most likely.

  • Surrogate is updated after each trial
  • Acquisition function (EI, UCB) picks next point
  • Converges in 20–50 trials vs 100s for random
  • Best for expensive evaluations (GPU hours)

Libraries: scikit-optimize, hyperopt, Optuna

Optuna Intermediate

Modern standard for HPO. Uses TPE with pruning — cancels poor trials early to save compute. Dashboard built in.

import optuna def objective(trial): lr = trial.suggest_float('lr', 1e-5, 1e-2, log=True) wd = trial.suggest_float('wd', 1e-6, 1e-3, log=True) lay = trial.suggest_int('layers', 2, 6) return train_model(lr, wd, lay) # returns val_loss study = optuna.create_study(direction='minimize') study.optimize(objective, n_trials=100) print(study.best_params)

Cross-Validation Strategies

k-Fold CV (k=5 or 10) — standard for tabular data. Rotate which fold is the validation set, average the k scores. More reliable than a single train/val split.

from sklearn.model_selection import StratifiedKFold, cross_val_score skf = StratifiedKFold(n_splits=5, shuffle=True, random_state=42) scores = cross_val_score(model, X, y, cv=skf, scoring='f1_macro') print(f'CV F1: {scores.mean():.3f} ± {scores.std():.3f}')
  • Stratified k-Fold — preserves class ratio in each fold (use for classification)
  • GroupKFold — prevents data from the same subject leaking across folds
  • TimeSeriesSplit — walk-forward validation for temporal data (never shuffle)
  • Nested CV — outer loop for performance estimate, inner for HPO; avoids optimistic bias
Adversarial & Edge Case Testing

Probe the model with inputs designed to break it — not to attack it, but to find blind spots before deployment does.

  • FGSM / PGD attacks — add imperceptible pixel noise that flips image classifier output
  • TextFooler / BERT-Attack — substitute synonyms to fool NLP models
  • Boundary samples — inputs near the decision boundary expose fragility
  • Out-of-distribution (OOD) — test on data from a different domain, season, or source
  • Null inputs — all-zeros, all-ones, empty strings, NaN-filled rows

Target: Graceful degradation, not crashes — model should return a low-confidence prediction rather than an exception.

Bias & Fairness Auditing

A model accurate on average may be systematically wrong for a subgroup. Always slice your test metrics by sensitive attributes.

  • Demographic parity — positive rate equal across groups
  • Equalized odds — equal TPR and FPR across groups
  • Calibration by subgroup — predicted probability matches actual frequency
# Slice test set by subgroup for grp, sub in test_df.groupby('gender'): preds = model.predict(sub[features]) f1 = f1_score(sub['label'], preds) print(f'{grp}: F1={f1:.3f}')

Tools: Fairlearn, IBM AI Fairness 360, What-If Tool

Latency & Throughput Profiling

A model that takes 800ms per prediction is unusable in a real-time product. Profile before you deploy.

Pre-Deploy Testing Checklist
  • Test set held out until final evaluation — never used during training or tuning
  • Cross-validation score and held-out score agree (±3% is normal; >5% gap = overfit)
  • Calibration curve plotted — predicted probabilities match real frequencies
  • OOD test set evaluated — how does accuracy degrade on unseen distributions?
  • Performance sliced by all known sensitive attributes
  • p50 and p99 latency measured under realistic batch sizes
  • Memory footprint profiled — model fits on target hardware
  • Null / malformed inputs return graceful output (no exceptions)
  • Model versioned and artifacts saved (weights + config + preprocessing)
  • Monitoring set up for data drift and prediction distribution shift

Common Pitfalls

These mistakes are responsible for most failed ML projects. Knowing them in advance is half the solution.

The central challenge of machine learning is finding the Goldilocks zone of model complexity:

  • A too-simple model (linear regression on a non-linear problem) has high bias — it makes systematic errors because it can't represent the true pattern.
  • A too-complex model (a 50-layer network on 500 samples) has high variance — it fits the training noise perfectly but fails on new data.
  • A just-right model learns the underlying signal and ignores noise — its training and validation metrics are close to each other.

This is formally called the Bias-Variance Tradeoff. Every modelling decision — architecture depth, regularisation, dataset size — moves you along this spectrum. The tools to find the sweet spot: cross-validation, learning curves, and regularisation.

Overfitting

Symptom: Training accuracy 98%, validation accuracy 72%. The model memorised the training set.

Fixes (in order of preference):

  1. Get more training data (most effective)
  2. Add Dropout (0.2–0.5 after each hidden layer)
  3. L2 regularisation (weight decay)
  4. Reduce model capacity (fewer layers / smaller hidden size)
  5. Data augmentation (images: flip, crop, colour jitter)
  6. Early stopping on validation loss
Never evaluate augmented data in your validation set — augmentation is training-only.
Underfitting

Symptom: Both training and validation accuracy are poor. The model has too little capacity or too little training.

Fixes:

  1. Increase model capacity (more layers / wider hidden sizes)
  2. Train longer (more epochs, check loss is still declining)
  3. Reduce regularisation (lower dropout, lower weight decay)
  4. Use a more expressive architecture (e.g., deep net vs. linear)
  5. Engineer better features — linear models underfit complex relationships
  6. Lower learning rate if loss is oscillating
Data Leakage

Future information leaks into training features, causing unrealistically high scores that collapse in production.

Common sources:

  • Scaling / normalisation fit on the full dataset before splitting
  • Target encoding computed on the whole dataset
  • Time-series: using future timestamps as features
  • Group leakage: same patient/user in train and test
Rule: Fit all preprocessing transformers on training data only. Apply (transform-only) to validation and test.
Vanishing & Exploding Gradients

In deep networks, gradients can shrink to zero (vanish) or grow uncontrollably (explode) during backprop, stalling training.

Vanishing fixes:

  • ReLU / Leaky ReLU / GELU instead of sigmoid/tanh
  • Residual connections (ResNet skip connections)
  • Batch Normalisation after linear layers
  • Careful weight initialisation (Kaiming / He init for ReLU)

Exploding fixes:

  • Gradient clipping: torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0)
  • Reduce learning rate
Class Imbalance

When 99% of examples are class 0, a model that predicts class 0 every time scores 99% accuracy — but is useless.

Fixes (ranked by effectiveness):

  1. Class weights in loss: CrossEntropyLoss(weight=class_weights)
  2. Oversampling minority: SMOTE (for tabular), duplicate with augmentation (images)
  3. Undersampling majority: Random or Tomek Links
  4. Threshold tuning: Default 0.5 threshold is wrong — tune on PR curve
  5. Use PR-AUC not accuracy as your metric

Feature Engineering

The single highest-leverage skill in applied ML. A mediocre model on great features beats a great model on raw data every time. "Garbage in, garbage out" is not a cliché — it's the root cause of most failed projects.

Numeric Features

Scaling — why it matters:

Algorithms that use distance (KNN, SVM) or gradient descent (neural nets, logistic regression) are sensitive to feature scale. A feature ranging 0–1,000,000 will dominate one ranging 0–1.

Feature Creation & Interactions

Creating new features from existing ones is often more valuable than tuning the model. Domain knowledge drives this.

  • Ratio features: price / sqft, clicks / impressions (CTR)

Data Pipelines

How data moves from raw source to model input — and how to make that process fast, reproducible, and leakage-free.

sklearn Pipeline — The Right Way

A Pipeline chains preprocessing steps and the model into one object. It guarantees no leakage (fit only on training data), enables cross-validation on the full pipeline, and makes deployment trivial — save one object, serve one object.

PyTorch Dataset & DataLoader

For deep learning, PyTorch separates data loading (Dataset) from batching and shuffling (DataLoader). This enables parallel loading with multiple CPU workers while the GPU trains.

Train / Val / Test Split Discipline
  • Train (70–80%) — model sees this during training
  • Validation (10–15%) — used for HPO and early stopping. Model never trains on it but you indirectly use it.
  • Test (10–15%) — touched exactly once, at the very end. Reports real-world performance.
If you tune hyperparameters on the test set, your test score is optimistic. The test set must be invisible until final evaluation.

For time series: always split by time — never shuffle. Future data cannot inform past predictions.

Data Augmentation

Artificially expand the training set by applying label-preserving transformations. Reduces overfitting. Never augment validation or test data.

Images:

  • Flip, crop, rotate, colour jitter, cutout, mixup
  • torchvision.transforms or albumentations

Text (NLP):

  • Back-translation, synonym swap, random deletion
  • nlpaug library

Tabular:

  • SMOTE (oversample minority class), Gaussian noise injection

Transfer Learning & Fine-Tuning

Training a model from scratch requires millions of labelled examples and days of GPU time. Transfer learning lets you start from a powerful pretrained model and adapt it to your task in hours — sometimes with only a few hundred examples.

The Core Idea

A model pretrained on a large dataset (ImageNet, Common Crawl, GitHub code) has already learned general representations — edges, textures, grammar, semantics. Your task just needs the final layers retrained to your specific labels.

ScenarioStrategy
Small dataset, similar domainFreeze all layers, retrain head only
Medium dataset, similar domainFine-tune last 2–4 layers + head
Large dataset, different domainFine-tune full model with low LR
Very different domainUse pretrained as init, train all
import torchvision.models as models # Load pretrained ResNet50 model = models.resnet50(weights='IMAGENET1K_V2') # Freeze all layers for param in model.parameters(): param.requires_grad = False # Replace final layer for your task (e.g. 5 classes) model.fc = nn.Sequential( nn.Dropout(0.3), nn.Linear(model.fc.in_features, 5) ) # Only model.fc parameters will be updated optimizer = torch.optim.AdamW( model.fc.parameters(), lr=1e-3)
Use a lower LR for pretrained layers than the new head — e.g., head LR = 1e-3, backbone LR = 1e-5. This prevents "catastrophic forgetting" of pretrained knowledge.

Large models can solve tasks with no (zero-shot) or very few (few-shot) examples — no gradient update needed.

  • Use RAG (Retrieval-Augmented Generation) for dynamic few-shot context from a vector database

LoRA — Parameter-Efficient Fine-Tuning (PEFT)

Fine-tuning a 7B parameter LLM requires ~56GB of VRAM — impractical for most. LoRA (Low-Rank Adaptation) freezes the original weights and injects small trainable rank-decomposition matrices into each layer. Only 0.1–1% of parameters are trained, but quality approaches full fine-tuning.

Explainability & Interpretability (XAI)

A model you can't explain is a model you can't trust in production — especially in regulated industries (finance, healthcare, legal). XAI bridges the gap between model accuracy and human understanding.

Interpretable vs Explainable — The Distinction

Intrinsically interpretable models:

Transparent by design — you can read the decision directly from the model itself.

  • Linear / Logistic Regression — each coefficient = effect of that feature on output
  • Decision Tree — follow the branch path to the leaf
  • Rule-based models — explicit IF-THEN logic
  • GAMs (Generalised Additive Models) — f(x) = f₁(x₁) + f₂(x₂) + … smooth, interpretable

Post-hoc explainability (black-box models):

Applied after training to explain individual predictions or global model behaviour. Works on any model.

  • SHAP — game-theoretic, model-agnostic, theoretically grounded
  • LIME — local linear approximation around one prediction
  • Permutation importance — shuffle a feature, measure accuracy drop
  • Attention maps — visualise which tokens/pixels the model focused on
  • Grad-CAM — gradient-weighted class activation maps for CNNs
SHAP — SHapley Additive exPlanations Intermediate

SHAP assigns each feature a Shapley value — its fair contribution to the prediction, averaging over all possible feature orderings. The only method that satisfies all four fairness axioms (efficiency, symmetry, dummy, additivity).

import shap # Tree-based models (fast, exact) explainer = shap.TreeExplainer(xgb_model) shap_vals = explainer.shap_values(X_test) # Summary plot — global feature importance shap.summary_plot(shap_vals, X_test) # Waterfall plot — explain one prediction shap.waterfall_plot( shap.Explanation(values=shap_vals[0], base_values=explainer.expected_value, data=X_test.iloc[0])) # For neural nets (slower, sample-based) explainer = shap.DeepExplainer(model, X_background)

Key plots: summary (global), waterfall (local), dependence (feature interaction), force plot (single prediction visualisation)

Time Series Forecasting

Time series data is ordered — past observations influence future ones. Standard ML assumptions (i.i.d. samples) break down. You need specialised models and validation strategies.

Never shuffle time series data. Splitting randomly creates data leakage — future values used to predict the past. Always split by time: train on earlier data, validate on later data.
Classical Statistical Models Beginner

ARIMA — AutoRegressive Integrated Moving Average:

from statsmodels.tsa.statespace.sarimax import SARIMAX # SARIMA(p,d,q)(P,D,Q,s) — with seasonality model = SARIMAX(y_train, order=(1,1,1), seasonal_order=(1,1,0,12)) # monthly seasonality fit = model.fit() forecast = fit.forecast(steps=12)

Parameters: p = AR order, d = differencing, q = MA order. Use AIC/BIC to select.

Best for: Univariate, stationary (or easily differenced), interpretable output required.

Walk-Forward Validation — The Right CV for Time Series

Standard k-fold is wrong for time series. Walk-forward (expanding window) validation mirrors real deployment — train on past, evaluate on future, expand the window and repeat.

from sklearn.model_selection import TimeSeriesSplit tscv = TimeSeriesSplit(n_splits=5, gap=0, # no gap between train/val test_size=90) # 90-day val window scores = [] for train_idx, val_idx in tscv.split(X): model.fit(X[train_idx], y[train_idx]) scores.append( rmse(y[val_idx], model.predict(X[val_idx]))) print(f'RMSE: {np.mean(scores):.2f} ± {np.std(scores):.2f}')
