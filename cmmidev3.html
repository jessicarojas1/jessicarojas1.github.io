📚 About CMMI v2.0 Level 3

CMMI v2.0 provides an updated, streamlined framework released in 2018. It reorganizes practices into Practice Areas and Practice Groups, adds Governance (GOV) and Implementation Infrastructure (II), and supports multiple constellations — Development and Services — through domain-specific Practice Areas layered on a shared Core.

ML2 (Managed): 12 Practice Areas — 9 Core (apply to all domains) + RDM (Development) + SD, IRP (Services). Work is planned, estimated, monitored, and controlled.

ML3 (Defined): 11 additional Practice Areas — 5 Core (DAR, CAR, OT, PAD, PCM) + 4 Development (TS, PI, VV, PR) + 2 Services (STSM, SSD). Processes are standardized and institutionalized at the organizational level.

Required Evidence by Practice Area

What appraisers need to see for each PA. ⭐ = Domain-specific (proven through execution in that domain).

🔶 Maturity Level 2 — Required Artifacts

CMConfiguration Management ML2
  • Version control system (Git, SVN, etc.)
  • Baseline definition — code, configs, docs under CM
  • Change control process with tickets tied to commits
  • Audit trail — who changed what, when
ESTEstimation ML2
  • Documented estimation method (story points, hours, parametric, etc.)
  • Historical data used from past sprints or projects
  • Estimates tied to actuals with variance tracking
GOVGovernance ML2
  • Defined roles and responsibilities (RACI or equivalent)
  • Management oversight evidence — reviews, approvals
  • Policy framework showing who enforces what
IIImplementation Infrastructure ML2
  • Documented tools and environments (Dev, Test, Prod)
  • CI/CD pipeline configurations and execution logs
  • Access provisioning records for required tools
  • Standards and templates available to teams
MPMManaging Performance & Measurement ML2
  • Defined metrics — velocity, defects, cycle time, etc.
  • Dashboards or reports showing metrics over time
  • Regular review cadence evidence (weekly/monthly records)
MCMonitor and Control ML2
  • Status tracking artifacts (Jira, Azure DevOps, etc.)
  • Variance management records — schedule/cost/scope deltas
  • Corrective action logs when things slip
PLANPlanning ML2
  • Project plans covering scope, schedule, and resources
  • Sprint and release planning artifacts
  • Dependencies identified and tracked
PQAProcess Quality Assurance ML2
  • Internal audit or QA check records
  • Evidence of noncompliance tracking
  • Corrective action tickets or logs
RDMRequirements Dev & Management ⭐ DEV CORE
  • Requirements documentation (user stories, specs, use cases)
  • Traceability matrix: Requirement → Design → Code → Test
  • Change management records for requirements
  • Stakeholder approval and validation records
RSKRisk and Opportunity Management ML2
  • Risk register with identified risks and opportunities
  • Risk scoring and prioritization evidence
  • Mitigation plans per risk item
  • Periodic risk review meeting records
SDService Delivery ⭐ SERVICES
  • Service agreements / SLAs defining services and service levels
  • Service performance reports vs. defined targets
  • Service delivery readiness checklists
  • Customer review meeting minutes addressing gaps
  • Service continuity and recovery plans
IRPIncident Resolution and Prevention ⭐ SERVICES
  • Incident classification criteria and severity matrix
  • Incident log / ticketing system records with resolution times
  • Stakeholder notification records
  • Root cause analysis reports for recurring incidents
  • Preventative action implementation records

🔶 Maturity Level 3 — Required Artifacts

CARCausal Analysis and Resolution ML3
  • Root cause analysis (RCA) documentation on defects/issues
  • Trend analysis showing recurring issue patterns
  • Preventative actions implemented and tracked to closure
DARDecision Analysis and Resolution ML3
  • Formal decision-making process documentation
  • Trade studies for tools, architecture, or vendor selection
  • Criteria-based evaluation matrices with scoring
OTOrganizational Training ML3
  • Training plans defined per role
  • Completion records (LMS exports or certificates)
  • Skills gap identification documentation
PRPeer Reviews ⭐ DEV CORE
  • Code review records (pull requests with actual comments)
  • Design review meeting notes or checklists
  • Review checklists or entry/exit criteria
  • Evidence reviews are actually happening — not just "LGTM"
PADProcess Asset Development ML3
  • Standard process documentation (SOPs, templates, playbooks)
  • Central repository (SharePoint, Confluence, etc.)
  • Versioning history of process documents
PCMProcess Management ML3
  • Organizational standard processes documented
  • Tailoring guidelines — how projects adapt processes
  • Governance records over process usage and compliance
PIProduct Integration ⭐ DEV CORE
  • Integration strategy document
  • Build process documentation and logs
  • Integration testing evidence and results
  • CI/CD pipelines executing integration builds
TSTechnical Solution ⭐ DEV CORE
  • System and software design documentation
  • Architecture decision records (ADRs)
  • Implementation artifacts aligned to requirements
  • Coding standards with evidence of adherence
VVVerification and Validation ⭐ DEV CORE
  • Test plans — unit, integration, system
  • Test execution results with pass/fail status
  • Requirement-to-test traceability (coverage matrix)
  • Acceptance validation records (customer or user sign-off)
STSMStrategic Service Management ⭐ SERVICES
  • Strategic service plan aligned to organizational goals
  • Service catalog describing services, SLA tiers, and constraints
  • Service capability and capacity assessments
  • Service strategy review records with improvement actions
SSDService System Development ⭐ SERVICES
  • Service system requirements document
  • Service system design and architecture documentation
  • Integration and verification plan for service system components
  • Service system verification and validation results
  • Service readiness review records
🔥 What Appraisers Will Actually Drill You On
🧠 Engineering Lifecycle — MAKE OR BREAK
RDM → TS → PI → VV → PR
Must be fully traceable and real. This chain is the entire point of the Development appraisal.
📊 Control & Discipline
PLAN, MC, MPM, RSK
Must show you are in control of your projects — not just doing work, but managing it.
🏢 Organizational Maturity
PAD, PCM, OT, PQA
Must show this isn't a one-off project — it's institutionalized across the organization.
🔍 Decision & Improvement
DAR, CAR
Must show you think and improve like an engineering org — data-driven decisions, root cause fixes.
🚨 You will FAIL if:
  • Processes exist but aren't actually used
  • No traceability — biggest failure point
  • No real peer reviews (just "LGTM" culture)
  • No metrics tied to actual decisions
  • No root cause analysis on defects
✅ You will PASS if:
  • Everything is connected — requirements trace to tests
  • Everything is repeatable — processes work on every project
  • Everything is evidenced in real work artifacts
